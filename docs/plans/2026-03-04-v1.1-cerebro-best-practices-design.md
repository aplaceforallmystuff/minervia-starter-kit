# Minervia v1.1 Design: Cerebro Best Practices Backport

**Date:** 2026-03-04
**Status:** Draft
**Audience:** Knowledge workers new to Claude Code (semi-technical)

## Problem

Minervia v1.0.0 ships a functional vault + skills setup, but it doesn't protect users from common AI assistant failure modes. These failure modes were discovered through months of daily Cerebro usage and are documented through real incidents:

- Claude runs `rm -rf` or `git push --force main` (catastrophic, unrecoverable)
- API keys pasted inline in Bash commands get permanently saved to `settings.json`
- Claude claims "tests pass" or "build succeeded" without actually running verification
- AI-generated content full of slop patterns gets published without detection
- Users accumulate duplicate skills/agents because nothing checks what already exists
- Claude fabricates URLs, statistics, or timestamps instead of checking sources

Cerebro has hooks, skills, and standing instructions that catch these. Minervia users shouldn't need to earn each scar themselves.

## Design Principles

1. **Approachable first** â€” new users shouldn't feel overwhelmed by safety rails
2. **Opt-in where possible** â€” hooks installed by default (they're invisible until triggered), skills offered as installer choices
3. **Simplified, not dumbed down** â€” Cerebro's "high" safety level, not "strict"; detection lists without the full 45-pattern editor mode
4. **Generic** â€” no Jim-specific domains, paths, project names, or platform references
5. **Documented origin** â€” each pattern links to the incident that created it, so users understand *why* it exists

## Component Design

### 1. Hooks (Installed by Default)

Hooks are invisible guardrails â€” users never interact with them unless something dangerous is about to happen. This makes them ideal for default installation.

#### 1a. `block-dangerous-commands.js`

**Source:** Cerebro's `~/.claude/hooks/block-dangerous-commands.js`
**Safety level:** `high` (not `strict` â€” avoids false positives for new users)

**What it blocks:**

| Level | Patterns | Action |
|-------|----------|--------|
| Critical | `rm -rf ~/`, `rm -rf /`, `dd` to disk device, fork bombs, `rm ~/.claude` | DENY with explanation |
| High | `curl \| bash` (RCE), `git push --force main`, `git reset --hard`, `git clean -f`, `chmod 777`, reading `.env` files, echoing `$SECRET`/`$TOKEN` vars, inline credentials (`API_KEY="real-value"`) | DENY with safer alternative |

**Generalization needed:**
- Remove Cerebro-specific inline credential patterns (`tskey-api-*` Tailscale keys)
- Keep generic patterns: `sk-*` (OpenAI/Stripe), `API_KEY=`, `Bearer` tokens
- Remove strict tier entirely (no `docker system prune` blocking â€” too aggressive for new users)

**Installed to:** `~/.claude/hooks/block-dangerous-commands.js`
**Hook config:** `PreToolUse` matcher on `Bash`

#### 1b. `protect-secrets.js`

**Source:** Cerebro's `~/.claude/hooks/protect-secrets.js`
**Safety level:** `high`

**What it blocks:**

| Level | Patterns | Action |
|-------|----------|--------|
| Critical | `.env` files, `.ssh/id_*`, `.aws/credentials`, `.kube/config`, `.pem`/`.key` files | DENY |
| High | `credentials.json`, `secrets.yaml`, service account keys, `.docker/config.json`, `.netrc`, `.npmrc`, `.pypirc` | DENY |
| High (Bash) | `cat .env`, `source .env`, `printenv`, `curl -d @.env` (exfiltration), `scp` of key files | DENY |

**Generalization needed:**
- Keep allowlist for `.env.example`, `.env.sample`, `.env.template`
- Remove any Cerebro-specific path patterns (none exist â€” already generic)
- Keep as-is; this hook is already fully generic

**Installed to:** `~/.claude/hooks/protect-secrets.js`
**Hook config:** `PreToolUse` matcher on `Read|Edit|Write|Bash`

#### Hook Installation Flow

The installer writes both hooks and adds the hooks configuration to the project's `.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command", "command": "node ~/.claude/hooks/block-dangerous-commands.js" }]
      },
      {
        "matcher": "Read|Edit|Write|Bash",
        "hooks": [{ "type": "command", "command": "node ~/.claude/hooks/protect-secrets.js" }]
      }
    ]
  }
}
```

**Important:** Hooks go in `~/.claude/settings.json` (global), not project-level. This protects the user across all projects, not just their vault.

#### 2a. `antislop` (Slim Version)

**Source:** Cerebro's `the-antislop` skill (45+ patterns, 6 categories, editor mode)
**Minervia version:** Detection + banned list only. No editor mode.

**What it includes:**
- The Horoscope Test ("Could anyone have written this, for anyone?")
- Banned phrases list (Tier 1: "delve", "tapestry", "game-changer", "paradigm shift", etc.)
- Structural pattern detection (staccato fragments, comparator sentences, emoji headers)
- Slop score calculation (Low/Medium/High/Critical)
- Report of findings with locations

**What it omits:**
- Editor mode (rewriting problems automatically) â€” requires voice profile context
- Full 45-pattern taxonomy â€” ship the top 20 most common patterns
- Integration with `draft-reviewer` agent â€” not shipping that agent in v1.1

**Invocation:** `/antislop` or "check this for AI slop"

#### 2b. `extract-wisdom`

**Source:** Cerebro's `extract-wisdom` skill
**Minervia version:** Fully generic â€” remove Cerebro-specific MCP server references

**What it includes:**
- 5 depth levels (Instant â†’ Comprehensive)
- Dynamic section generation based on content type
- Level 3 tone rules ("smart friend over coffee, not book report")
- Support for YouTube URLs, article URLs, file paths, pasted text

**Generalization needed:**
- Replace `YouTube transcript MCP` reference with generic "use WebFetch on transcript services or paste transcript directly"
- Remove references to Cerebro vault locations for saving output
- Add generic "save to your vault" suggestion

**Invocation:** `/extract-wisdom [URL or text]` or "extract wisdom from this"

#### 2c. `creation-guard` (Simplified)

**Source:** Cerebro's `creation-guard` skill
**Minervia version:** Simplified ecosystem search â€” no capability inventory file, no remote server checks

**What it includes:**
- Search existing `~/.claude/skills/`, `~/.claude/agents/`, `~/.claude/commands/` for overlap
- Present findings with recommendation: PROCEED / EXTEND / BLOCK
- Trigger on intent to create new skills, agents, or commands

**What it omits:**
- Capability inventory file consultation (Cerebro-specific)
- Remote infrastructure checks (Cerebro-specific)
- `~/Dev/` project scanning (too broad for starter kit)
- COMPOSE and ITERATE verdicts (simplify to 3 outcomes)

**Invocation:** Automatic â€” triggered by creation intent phrases in conversation

#### 2d. `systematic-debugging`

**Source:** Cerebro's `systematic-debugging` skill
**Minervia version:** As-is â€” already fully generic

**What it includes:**
- Four phases: Root Cause Investigation â†’ Pattern Analysis â†’ Hypothesis & Testing â†’ Implementation
- Iron law: "No fixes without root cause investigation first"
- "3 fixes failed = question the architecture" rule
- Rationalization prevention table

**No generalization needed.** This skill contains zero Cerebro-specific content.

**Invocation:** Automatic â€” triggered by debugging scenarios

#### 2e. `verification-before-completion`

**Source:** Cerebro's `verification-before-completion` skill
**Minervia version:** Core gate function only â€” omit Verification Rehearsal technique

**What it includes:**
- The Iron Law: "No completion claims without fresh verification evidence"
- The Gate Function (5-step: Identify â†’ Run â†’ Read â†’ Verify â†’ Claim)
- Common failures table (what "tests pass" actually requires)
- Red flags and rationalization prevention

**What it omits:**
- Verification Rehearsal technique (pre-build simulation) â€” too advanced for v1.1
- Anti-Criteria Checklist â€” adds complexity without proportional value for beginners
- Cerebro integration references

**Invocation:** Automatic â€” triggered when Claude is about to claim completion

### 3. CLAUDE.md Template Additions

These are standing instructions added to the generated CLAUDE.md. They cost nothing (no new files, no new skills) and prevent real problems.

#### 3a. Permissions Array Warning

```markdown
## Security Note

When you approve a Bash command, Claude Code saves the full command text to
`~/.claude/settings.json`. This means inline credentials get permanently stored:

- BAD: `API_KEY="sk-real-key" node script.js` â€” saves the key to settings forever
- GOOD: Export the credential first (`export API_KEY="value"`), then run the command without it

If this happens accidentally, edit `~/.claude/settings.json` to remove the entry
and rotate the exposed credential.
```

**Why:** This is a real footgun that nobody documents. Cerebro discovered it through an actual credential exposure incident.

#### 3b. Date Validation Rule

```markdown
## Date Validation

The system clock is the only source of truth for dates. Never trust day-of-week
from file content â€” verify with `date` command if accuracy matters.
```

#### 3c. Verification Before Completion

```markdown
## Verification Rule

Before claiming work is complete, fixed, or passing â€” run the verification command
and show the output. "Should work" is not evidence. Evidence is evidence.
```

#### 3d. Uncertainty Protocol Enhancement

Add to the existing Uncertainty Protocol section:

```markdown
- **Numbers and statistics:** Verify from primary sources before writing them.
  Generic is better than fabricated. "I can't verify this" beats a plausible guess.
```

#### 3e. Editors Not Critics

```markdown
## Review Principle

When reviewing or auditing content, apply fixes directly â€” don't just recommend them.
"Consider changing X to Y" wastes time. Change X to Y and explain why.
```

### 4. Installer Changes

#### 4a. New Questions

Add after existing question 5 (preferences):

**Question 6:** "Would you like AI writing assistance skills?"
- Options: `Yes (Recommended)` / `No`
- If yes: install `antislop`, `extract-wisdom`
- Default: Yes

**Question 7:** "Would you like defensive development skills?"
- Options: `Yes (Recommended)` / `No`
- If yes: install `creation-guard`, `systematic-debugging`, `verification-before-completion`
- Default: Yes

Both questions use `ask_choice` with the recommended option first.

#### 4b. Hooks Installation Step

After skill installation, add a new step:

```
Installing safety hooks...
  âś“ block-dangerous-commands.js
  âś“ protect-secrets.js
  âś“ hooks configured in ~/.claude/settings.json
```

Hooks are always installed (not gated by a question) because:
- They're invisible until triggered
- They prevent catastrophic data loss
- There's no UX cost to having them
- New users are the most likely to accidentally run dangerous commands

#### 4c. State Tracking

Add to `~/.minervia/state.json`:
- `hooks_installed: true/false`
- `writing_skills_installed: true/false`
- `dev_skills_installed: true/false`

This enables the update system to know which optional components to update.

## Migration Path

For existing v1.0.0 users running `/minervia:update`:

1. Hooks are installed automatically (new step in update flow)
2. New skills are offered: "v1.1 includes new skills. Install writing assistance skills? (Y/n)"
3. CLAUDE.md additions are appended (checksum detects user hasn't modified those sections)
4. If user has customized CLAUDE.md, show diff and offer merge options (existing behavior)

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| Hooks block legitimate commands (false positives) | Using `high` level, not `strict`. Critical tier is unambiguous. High tier patterns are genuinely dangerous. |
| New questions make installer too long | 7 questions total (was 5). Both new ones are simple Yes/No with recommended defaults. |
| Skills feel overwhelming | Gated behind opt-in questions. Not mentioned unless installed. |
| CLAUDE.md becomes too long | 5 additions total ~25 lines. Template is currently 81 lines. Still under 110. |
| Update flow confusion | State tracking distinguishes what's installed. Update only touches installed components. |
